# AN/BRU-370 Decision Log

Decisions that aren't obvious from the code, commit messages, or CLAUDE.md. Each entry captures what was decided, why, and what was rejected.

| Date | Decision | Why | Alternatives Rejected |
|---|---|---|---|
| 2026-05-31 | Wi-Fi hostname set before `WiFi.mode()` | Hostname set after `mode()` doesn't propagate reliably to the DHCP lease on all routers | Set hostname after `WiFi.mode()` (default order — unreliable) |
| 2026-05-31 | 3-second non-blocking USB settle before `startConnect()` | ESP32-S3 shares a PMU between USB-OTG and Wi-Fi. Calling `WiFi.begin()` while USB enumeration is in-flight causes intermittent Wi-Fi failures (~2/5 boots) | No delay (intermittent failure); blocking `delay(3000)` in `setup()` (breaks USB enumeration) |
| 2026-06-01 | USB PID fixed at `0x370A` | Windows and DCS cache HID device identity by PID. Changing PID breaks all existing axis/button bindings in DCS controls menu and forces re-binding | No fixed PID / auto-assigned PID |
| 2026-06-01 | Encoder character scroll for Manual Wi-Fi entry | Serial port is inaccessible when the device is in USB HID composite mode — `mouhid.sys` and the HID driver claim the USB interfaces, leaving no visible COM port | Serial/USB serial entry (broken in composite mode) |
| 2026-06-05 | HID digitizer uses `Usage(Pen)` not `Usage(Mouse)` | Windows `mouhid.sys` applies a ~16 px minimum-displacement dead zone to all absolute `Usage(Mouse)` reports. This filter cannot be disabled by any user setting. `hidpen.sys` (digitizer driver) maps absolute coordinates 1:1 | `Usage(Mouse)` absolute mode (16 px dead zone makes fine positioning impossible) |
| 2026-06-06 | Pin Label calibration uses absolute screen coordinates (`lbX2`/`lbY2`) | Original `lbX`/`lbY` stored a relative delta from map center. When the user re-calibrated map center, the label position silently shifted. Absolute coordinates are independent | Relative-delta approach (inherited stale offsets; old NVS keys left as harmless orphans) |
| 2026-06-07 | DCS-BIOS auto-starts after manual Connect action | Without this, a successful manual reconnect left the device in a state where the Wi-Fi was live but DCS-BIOS never started — the user had to reboot to get cockpit data | Require reboot after manual connect |
| 2026-06-08 | Stores Config uses a smart-send state machine, not a simple toggle | A simple toggle caused oscillation: the device sent a command → DCS exported the new switch state → the device saw the "old" state still set → sent the opposite command → loop. The state machine locks the target on first press and monitors for confirmation | Simple toggle (oscillation); user-held button approach (requires sustained press in a stress situation) |
| 2026-06-08 | `LIGHT_STORES_CONFIG` check placed before the `mcConfirmed` gate | Addresses `0x4478` (STORES CONFIG light) and `0x447A` (MASTER CAUTION) are adjacent in the DCS-BIOS export stream and arrive in the same UDP frame. When MC and the Stores Config light both clear simultaneously, checking the SC light inside the `mcConfirmed` block means the block is never entered (MC already false) | Check inside `mcConfirmed` block (missed the SC clear event in the same frame) |
| 2026-06-09 | SC_SHOW_TIMING state removed; timing display not shipped | The state added complexity and the 4-second pause was disruptive during real sorties. Timing data collection can happen via OLED diagnostics if needed in the future | SC_SHOW_TIMING as fifth state (removed in refactor f667339) |
| 2026-06-14 | AP PITCH/ROLL switches and HID Gamepad interface removed | Target user no longer requires these controls; DCS-BIOS provides better alternatives for in-game switch and axis binding. Implementation preserved on `archive/ap-switches-gamepad` branch. | Conditional compilation (#ifdef) — adds clutter for no planned re-use timeline |
| 2026-06-15 | OTA delivered via GitHub raw HTTPS, not ArduinoOTA or local BLE | No local server infrastructure required; GitHub releases host the binary; `OTA_MANIFEST_URL` in `config.h` is the only configuration needed. ArduinoOTA requires a local network upload tool running on the host PC at flash time. | ArduinoOTA (requires host tool active during update); BLE binary transfer (too slow for a full firmware image) |
| 2026-06-16 | Boot status 6-phase diagnostic grid replaced the splash screen | Users had no information about which Wi-Fi phase was failing. The grid shows RF / SSID / ETH / IP / DNS / DCS as ✓ or ✗ in real time, making connection failures immediately diagnosable without serial output. | Retained splash (no diagnostic value); simple "Connecting…" text (no phase breakdown) |
| 2026-06-16 | Wi-Fi credentials moved to NVS-only; compiled-in defaults removed | A committed `config.h` with credentials would appear in git history and is a security risk when the repo is public. NVS isolates credentials from firmware images; OTA updates cannot overwrite them. | Compiled-in defaults with `.gitignore` (credentials still in repo history); BLE-only first-time setup (required BLE phone for every fresh flash) |
| 2026-06-16 | Wi-Fi menu redesigned: Secrets submenu holds credential management | The old menu mixed connection actions (Connect, Manual entry) with destructive credential operations at the same level. Moving Zeroize and BLE TERM behind a Secrets submenu prevents accidental credential changes during normal use. Manual encoder entry was removed — BLE terminal is faster and less error-prone. | Keep Manual entry alongside Connect (one-level flat menu); no Secrets gate |
| 2026-06-16 | OTA version encoding changed from BCD to plain decimal integer | BCD encoding of `v0.30` would be `0x0030 = 48 decimal`. The next version after `v0.29` is `30` in decimal, not `48`. This arithmetic gap meant the manifest and device comparison would fail at v0.30. Switching to plain decimal (`FIRMWARE_VERSION_INT`) avoids the representation ambiguity entirely. A bridge version `v0.50` was cut to move past the BCD gap cleanly. | Continue BCD (version comparison breaks at v0.30+); use semver (overkill for a single-device single-binary OTA) |
| 2026-06-17 | Aircraft Status screen as the default home when DCS is connected | The Macro Menu is not useful during a DCS sortie unless a map action is needed. Showing fuel, CMDS counts, and ECM state at a glance improves situational awareness at the panel without any encoder input. The Macro Menu is still reachable with a turn or short press. | Always default to Macro Menu (hides useful cockpit data); add a separate "status" macro item (adds menu clutter) |
| 2026-06-17 | CMDS strings displayed as raw DCS output, not re-parsed | DCS exports chaff/flare count as a 4-character ASCII string (e.g. `"  60"`, `"Lo10"`). Re-parsing it to extract a number would require handling the `Lo` prefix, right-alignment, and possible future DCS format changes. Displaying the raw string is always correct and zero-maintenance. The `Lo` prefix is detected for the blink effect only. | Parse to integer (breaks on `Lo` prefix; fragile to DCS format changes); omit low-count indication (no blink cue when running low) |
| 2026-06-17 | `safeRestart()` deauths from AP before every reboot | Calling `ESP.restart()` without disconnecting Wi-Fi leaves the router session open. On the next boot, the ESP32 tries to re-associate and the router responds with `WIFI_REASON_AUTH_EXPIRE` (error 2), causing the first boot attempt to fail. `WiFi.mode(WIFI_OFF)` + 500 ms delay ensures the router releases the session cleanly. | `ESP.restart()` directly (transient error 2 on next boot); `WiFi.disconnect()` only (driver state not fully cleared) |
