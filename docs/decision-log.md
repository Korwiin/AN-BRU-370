# AN/BRU-370 Decision Log

Decisions that aren't obvious from the code, commit messages, or CLAUDE.md. Each entry captures what was decided, why, and what was rejected.

| Date | Decision | Why | Alternatives Rejected |
|---|---|---|---|
| 2026-05-31 | Wi-Fi hostname set before `WiFi.mode()` | Hostname set after `mode()` doesn't propagate reliably to the DHCP lease on all routers | Set hostname after `WiFi.mode()` (default order — unreliable) |
| 2026-05-31 | 3-second non-blocking USB settle before `startConnect()` | ESP32-S3 shares a PMU between USB-OTG and Wi-Fi. Calling `WiFi.begin()` while USB enumeration is in-flight causes intermittent Wi-Fi failures (~2/5 boots) | No delay (intermittent failure); blocking `delay(3000)` in `setup()` (breaks USB enumeration) |
| 2026-06-01 | USB PID fixed at `0x370A` | Windows and DCS cache HID device identity by PID. Changing PID breaks all existing axis/button bindings in DCS controls menu and forces re-binding | No fixed PID / auto-assigned PID |
| 2026-06-01 | Encoder character scroll for Manual Wi-Fi entry | Serial port is inaccessible when the device is in USB HID composite mode — `mouhid.sys` and the HID driver claim the USB interfaces, leaving no visible COM port | Serial/USB serial entry (broken in composite mode) |
| 2026-06-05 | HID digitizer uses `Usage(Pen)` not `Usage(Mouse)` | Windows `mouhid.sys` applies a ~16 px minimum-displacement dead zone to all absolute `Usage(Mouse)` reports. This filter cannot be disabled by any user setting. `hidpen.sys` (digitizer driver) maps absolute coordinates 1:1 | `Usage(Mouse)` absolute mode (16 px dead zone makes fine positioning impossible) |
| 2026-06-06 | Pin Label calibration uses absolute screen coordinates (`lbX2`/`lbY2`) | Original `lbX`/`lbY` stored a relative delta from map center. When the user re-calibrated map center, the label position silently shifted. Absolute coordinates are independent | Relative-delta approach (inherited stale offsets; old NVS keys left as harmless orphans) |
| 2026-06-07 | DCS-BIOS auto-starts after manual Connect action | Without this, a successful manual reconnect left the device in a state where the Wi-Fi was live but DCS-BIOS never started — the user had to reboot to get cockpit data | Require reboot after manual connect |
| 2026-06-08 | Stores Config uses a smart-send state machine, not a simple toggle | A simple toggle caused oscillation: the device sent a command → DCS exported the new switch state → the device saw the "old" state still set → sent the opposite command → loop. The state machine locks the target on first press and monitors for confirmation | Simple toggle (oscillation); user-held button approach (requires sustained press in a stress situation) |
| 2026-06-08 | `LIGHT_STORES_CONFIG` check placed before the `mcConfirmed` gate | Addresses `0x4478` (STORES CONFIG light) and `0x447A` (MASTER CAUTION) are adjacent in the DCS-BIOS export stream and arrive in the same UDP frame. When MC and the Stores Config light both clear simultaneously, checking the SC light inside the `mcConfirmed` block means the block is never entered (MC already false) | Check inside `mcConfirmed` block (missed the SC clear event in the same frame) |
| 2026-06-09 | SC_SHOW_TIMING state removed; timing display not shipped | The state added complexity and the 4-second pause was disruptive during real sorties. Timing data collection can happen via OLED diagnostics if needed in the future | SC_SHOW_TIMING as fifth state (removed in refactor f667339) |
| 2026-06-14 | AP PITCH/ROLL switches and HID Gamepad interface removed | Target user no longer requires these controls; DCS-BIOS provides better alternatives for in-game switch and axis binding. Implementation preserved on `archive/ap-switches-gamepad` branch. | Conditional compilation (#ifdef) — adds clutter for no planned re-use timeline |
| 2026-06-15 | OTA delivered via GitHub raw HTTPS, not ArduinoOTA or local BLE | No local server infrastructure required; GitHub releases host the binary; `OTA_MANIFEST_URL` in `config.h` is the only configuration needed. ArduinoOTA requires a local network upload tool running on the host PC at flash time. | ArduinoOTA (requires host tool active during update); BLE binary transfer (too slow for a full firmware image) |
| 2026-06-16 | Boot status 6-phase diagnostic grid replaced the splash screen | Users had no information about which Wi-Fi phase was failing. The grid shows RF / SSID / ETH / IP / DNS / DCS as ✓ or ✗ in real time, making connection failures immediately diagnosable without serial output. | Retained splash (no diagnostic value); simple "Connecting…" text (no phase breakdown) |
| 2026-06-16 | Wi-Fi credentials moved to NVS-only; compiled-in defaults removed | A committed `config.h` with credentials would appear in git history and is a security risk when the repo is public. NVS isolates credentials from firmware images; OTA updates cannot overwrite them. | Compiled-in defaults with `.gitignore` (credentials still in repo history); BLE-only first-time setup (required BLE phone for every fresh flash) |
| 2026-06-16 | Wi-Fi menu redesigned: Secrets submenu holds credential management | The old menu mixed connection actions (Connect, Manual entry) with destructive credential operations at the same level. Moving Zeroize and BLE TERM behind a Secrets submenu prevents accidental credential changes during normal use. Manual encoder entry was removed — BLE terminal is faster and less error-prone. | Keep Manual entry alongside Connect (one-level flat menu); no Secrets gate |
| 2026-06-16 | OTA version encoding changed from BCD to plain decimal integer | BCD encoding of `v0.30` would be `0x0030 = 48 decimal`. The next version after `v0.29` is `30` in decimal, not `48`. This arithmetic gap meant the manifest and device comparison would fail at v0.30. Switching to plain decimal (`FIRMWARE_VERSION_INT`) avoids the representation ambiguity entirely. A bridge version `v0.50` was cut to move past the BCD gap cleanly. | Continue BCD (version comparison breaks at v0.30+); use semver (overkill for a single-device single-binary OTA) |
| 2026-06-17 | Aircraft Status screen as the default home when DCS is connected | The Macro Menu is not useful during a DCS sortie unless a map action is needed. Showing fuel, CMDS counts, and ECM state at a glance improves situational awareness at the panel without any encoder input. The Macro Menu is still reachable with a turn or short press. | Always default to Macro Menu (hides useful cockpit data); add a separate "status" macro item (adds menu clutter) |
| 2026-06-17 | CMDS strings displayed as raw DCS output, not re-parsed | DCS exports chaff/flare count as a 4-character ASCII string (e.g. `"  60"`, `"Lo10"`). Re-parsing it to extract a number would require handling the `Lo` prefix, right-alignment, and possible future DCS format changes. Displaying the raw string is always correct and zero-maintenance. The `Lo` prefix is detected for the blink effect only. | Parse to integer (breaks on `Lo` prefix; fragile to DCS format changes); omit low-count indication (no blink cue when running low) |
| 2026-06-17 | ~~`safeRestart()` deauths from AP before every reboot~~ — **superseded 2026-06-29**, see below | Calling `ESP.restart()` without disconnecting Wi-Fi leaves the router session open. On the next boot, the ESP32 tries to re-associate and the router responds with `WIFI_REASON_AUTH_EXPIRE` (error 2), causing the first boot attempt to fail. `WiFi.mode(WIFI_OFF)` + 500 ms delay ensures the router releases the session cleanly. | `ESP.restart()` directly (transient error 2 on next boot); `WiFi.disconnect()` only (driver state not fully cleared) |
| 2026-06-29 | Custom WiFi deauth sequence removed from `safeRestart()` and OTA restart; bare `ESP.restart()` only | Every official ESP32 reference (ArduinoOTA, HTTPUpdate, CockpitOS) uses bare `ESP.restart()` with no prior WiFi teardown. The custom deauth sequence (`WiFi.disconnect()` + `WiFi.mode(WIFI_OFF)` + delays) was built to work around AUTH_EXPIRE failures that were actually caused by buggy WPA3 negotiation in arduino-esp32 (now fixed via PMF disable, see below). The deauth-then-reauth cycle, combined with the arduino-esp32 auto-reconnect loop also sending its own deauth+reconnect on every AUTH_EXPIRE, was bombarding the AP and resetting its stale-session cleanup timer — actively causing the reconnect failures it was meant to prevent. Removing it improved OTA restart reconnect reliability. | Keep the custom deauth (caused AUTH_EXPIRE bombardment); add more deauth/delay tuning (treating the symptom, not the cause) |
| 2026-06-29 | `esp_wifi_disable_pmf_config()` after every `WiFi.begin()` forces WPA2, working around arduino-esp32 WPA3/PMF bugs | The AP (Eero) operates in WPA2/WPA3 transition mode. arduino-esp32 has open, unfixed GitHub issues (#6767, #6843, #7605) where `WiFi.begin()` hardcodes `pmf_cfg.capable=true`, triggering WPA3 SAE negotiation that intermittently fails with AUTH_EXPIRE. Disabling PMF on the client side prevents WPA3 SAE from being negotiated at all — the AP falls back to WPA2, which doesn't have the SAE session-teardown timing problem. Must be re-applied in the DISCONNECTED event handler because arduino-esp32's internal `first_connect` auto-retry re-calls `WiFi.begin()`, which re-enables PMF. | Disable PMF only once at boot (insufficient — first_connect retry re-enables it); wait for an arduino-esp32 fix (issues open since 2022, unresolved) |
| 2026-06-29 | Boot WiFi retry changed from 3-attempt-with-backoff to indefinite auto-reconnect (`startWifi()` called once, `setAutoReconnect(true)` from the start) | The 3-attempt system (with WIFI_OFF cycling between attempts) gave up after ~66 seconds — well short of the 60-90 second window Eero's WPA3/WPA2 session sometimes needs to clear after an ungraceful restart (e.g. power loss). Indefinite retry with the arduino-esp32 stack's built-in reconnect lets the device keep trying without an artificial give-up point, matching mainstream ESP32 practice (CockpitOS, official examples). | 3-attempt retry with longer backoff (still has a give-up point; user must manually intervene); shorter backoff (worse — re-triggers the AP bombardment problem) |
| 2026-06-29 | `WiFi.mode(WIFI_OFF)` removed from the boot and routine-reconnect paths; retained only in `reconnectFull()` for long (≥60 s) outages | Cycling the radio off/on between connection attempts sends management frames that reset the AP's stale-session cleanup timer — directly causing the AUTH_EXPIRE bombardment problem above. For short/transient drops, staying in `WIFI_STA` and re-issuing `WiFi.begin()` avoids this. For confirmed long outages (60s+, detected by the runtime watchdog), the AP's session has certainly expired, so a full radio reset is safe and sometimes necessary to clear ESP32-side driver state. | Always cycle WIFI_OFF (causes bombardment); never cycle WIFI_OFF (long outages sometimes need a full driver reset to recover) |
| 2026-06-29 | Boot retry on `WIFI_REASON_TIMEOUT` (39, "Conn timeout") gets its own 30 s app-level retry, separate from arduino-esp32's built-in reconnect | Reason 39 is not in arduino-esp32's `_isReconnectableReason()` list, so the stack's auto-reconnect never retries it — after the one-time `first_connect` retry also fails with reason 39, the device would otherwise be stuck indefinitely with no further attempts. The app-level retry (`s_bootRetryAt`, resets `s_bootStarted=false` after 30 s) is a deliberate gap-fill for reason codes the stack silently refuses to handle. | Rely solely on arduino-esp32's built-in reconnect (silently stalls on reason 39); patch arduino-esp32's reconnectable-reasons list (would require a forked core) |
| 2026-06-29 | WiFi made a mandatory core function; the Settings → WiFi → DISABLE/ENABLE toggle was removed | DCS-BIOS connectivity is the device's primary purpose; "WiFi disabled" was a degraded mode (HID-macro-pad-only) that added state-machine complexity (`s_wifiEnabled`, `s_wifiCancelled` flags threaded through boot, watchdog, and menu logic) for a configuration nobody used in practice. `homeMode()` now unconditionally returns `BOOT_STATUS` when DCS is not connected (was `MACRO_MENU`), making the boot/status screen the single "not ready" state regardless of cause (initial boot, mid-session WiFi drop, or DCS disconnect). | Keep the toggle (unused complexity); keep MACRO_MENU as a no-DCS fallback (let users operate without WiFi — not a real use case for this device) |
| 2026-06-29 | DCS-BIOS commands sent via unicast to the sender IP captured from the last received export packet, not UDP broadcast to `255.255.255.255` | `WiFiUDP::beginPacket()` to a broadcast address from a multicast-joined socket silently fails — `SO_BROADCAST` is not set on a socket opened with `beginMulticast()`, and lwIP drops the broadcast send with no error surfaced to the caller. DCS-BIOS's `UDPListener` binds to `0.0.0.0:7778` (confirmed in `BIOSConfig.lua`), so unicast to the PC's actual IP works without needing the broadcast flag at all. The PC's IP is available for free — it's the source address of every multicast export packet received. | Set `SO_BROADCAST` on the socket (would require a second socket or low-level lwIP calls AsyncUDP/WiFiUDP don't expose); keep broadcast and silently lose all outbound commands (the bug this fixed) |
| 2026-06-29 | DCS-BIOS UDP transport migrated from `WiFiUDP` (polled in `loop()`) to `AsyncUDP` with a portMUX-protected ring buffer | `WiFiUDP::parsePacket()` polling ties packet reception to `loop()`'s tick rate — a slow tick (display redraw, encoder debounce) can miss or delay packets. `AsyncUDP`'s `onPacket` callback fires on Core 0 the instant a packet arrives, independent of `loop()` timing; Core 1 drains the ring buffer at its own pace. Matches the architecture used by CockpitOS (the reference ESP32 DCS-BIOS framework) for the same reason. | Stay on `WiFiUDP` (loop-coupled, can miss frames under load); use a FreeRTOS queue instead of a ring buffer (more overhead for no benefit at this packet rate) |
| 2026-06-29 | Boot/status screen redesigned from a 6-checkbox phase grid (RF/SSID/ETH/IP/DNS/DCS) to 3 indicators (WiFi/IP/DCS) showing live data instead of checkmarks — **supersedes 2026-06-16 grid entry** | The 6-phase grid showed binary pass/fail with no diagnostic depth — RF and ETH always flip together with SSID (same event), and DNS is synchronous with IP (not a separate async step), so 3 of the 6 checkboxes carried no independent information. The replacement shows the WiFi slot as live auth mode (`WPA2`/`WPA3`) once associated and the IP slot as the actual last octet once DHCP completes — same screen real estate, more useful data. A status line below shows exactly one of three states (`Connecting ...`, `<reason> Retry N` with a live 30 s countdown, `Waiting for DCS ...`), replacing a static-text/animated-spinner mix with a single consistent style. | Keep the 6-checkbox grid (low information density per pixel); add a 4th line for diagnostics instead of replacing checkmarks with data (screen is already at its 4-line limit on the 128×32 OLED) |
| 2026-06-29 | OTA releases must always be built from the `esp32s3_supermini_release` PlatformIO environment, never the default debug environment | A debug build was shipped via OTA at v0.84 before this was caught — debug builds leave `Serial.begin()` active (gated by `#ifndef RELEASE_BUILD`), and on this ESP32-S3's shared USB-OTG port, active CDC Serial conflicts with USB HID (keyboard + digitizer), the device's primary I/O. The release environment defines `RELEASE_BUILD=1`, disabling Serial entirely. This had been silently wrong for several prior OTA releases (v0.82, v0.83) with no user-visible symptom until HID stopped working, since the user happened to not be testing HID at the time (using a USB-C battery pack, not the PC's USB port). | Detect HID/Serial conflict at runtime and disable Serial automatically (adds runtime complexity for a build-time decision); document the requirement only (proven insufficient — it was already documented in CLAUDE.md and still got missed once) |
| 2026-07-17 | Never pump `LvglPort::loop()`/`lv_timer_handler` from inside an LVGL event callback; use `lv_timer_create` for countdowns and `lv_refr_now()` for forced paints in blocking phases | LVGL 9's `lv_timer_handler` recursion guard makes nested calls a silent no-op (no render, no touch) — shipped broken Reboot/USB-Flash countdowns on the anbru430 branch before being caught in final review | Pumping the loop from the callback anyway (recursion guard makes it a silent no-op); blocking on `lv_refr_now()` in a countdown loop (paints, but touch input stays dead so Cancel is unreachable) |
| 2026-07-17 | LVGL heap moved to PSRAM (`LV_USE_STDLIB_MALLOC=LV_STDLIB_CUSTOM` + `src/anbru430/lv_mem_psram.c` on `heap_caps_malloc(..., MALLOC_CAP_SPIRAM)`) | The 55 KB internal-heap starvation incident: with the full parity UI (all pages + alerts) built at boot plus 64 KB bounce buffers and WiFi, internal free heap sat at ~55 KB — below the ~45-50 KB mbedTLS needs — so every OTA HTTPS check failed instantly (`a1:-1/0s`) | Shrinking the bounce buffers back down (loses the v0.05 flicker fix); reverting to the builtin TLSF pool in internal RAM (the 3a static-pool failure) |
